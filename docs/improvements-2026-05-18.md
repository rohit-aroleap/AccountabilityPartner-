## Summary

- Dashboard ideas: **96** across **12** categories  
- Prompt engineering ideas: **91** across **14** categories  
- Top emergent themes:
  1. **Context gap** — Customer notes, onboarding answers, and personality settings are captured but never injected into manual draft prompts (`js/prompt.js`), creating a split between what the worker knows and what the dashboard knows.  
  2. **Multi-customer operations are underdeveloped** — The customer list has no search, sort, filter, or batch actions, which becomes painful as the customer count grows beyond 20.  
  3. **Eval and prompt iteration discipline** — The Sandbox is excellent but there's no golden test set, no prompt diff-on-save, no rating aggregation view, making systematic prompt improvement harder than it needs to be.

## Ambitious bets (L effort but transformational)

1. **#54 — Batch daily review mode**: A single-screen flow to review all pending drafts across all customers without switching chat panes. Would make the morning workflow 5× faster when managing 20+ customers. Requires a new modal with multi-customer Firebase subscriptions.

2. **P31 — Few-shot library (Firebase-backed)**: Operator saves the best drafts they've actually sent. Future prompt calls inject the top 3 as few-shot examples. Creates a self-improving prompt that gets better with every good message sent.

3. **P87 — A/B prompt mode**: Deploy prompt variant A to half the customers, B to the other half. Compare acceptance rates and 👍 rates over 2 weeks. The only rigorous way to know if a prompt change is actually better. Requires `promptsB` in Firebase and variant routing in the worker.

4. **#44 — Auto-flag prompt quality warning**: An automated quality gate that detects when drafts are getting worse (based on real ratings) and surfaces it before the operator has to notice. Closes the feedback loop between rating data and prompt tuning.

5. **#55 (Mobile responsive layout + PWA)**: The dashboard is currently unusable on a phone. Given that most of the operator's customers are on WhatsApp (mobile-first), the operator is likely checking in from a phone too. A mobile-friendly layout + PWA install would make the dashboard portable.

## Top 10 to ship next

Rank-ordered by impact/effort/current pain, mixing both lists.

1. **P1 — Personality in manual drafts** (S effort, XL impact): The most jarring inconsistency in the product. Auto-sends use the customer's chosen personality; dashboard 💪 drafts don't. Operators who notice this lose trust in the manual-draft feature. One `if` block in `getSystemForCustomer`.

2. **P3 — Customer notes in prompts** (S effort, XL impact): The notes field exists, operators are filling it ("shoulder injury", "prefers evenings"), but the AI never sees it. This is a broken promise. Fix `buildFerraPrompt` and `buildGymPrompt` to include notes.

3. **#23 + #24 — Search + sort in customer list** (S effort each, L impact): With > 15 customers, navigating the sidebar is already slow. Search by name and sort by urgency/habit score are day-1 features for any list UI. Both implementable in `js/customers.js` in a few hours.

4. **P29 — 3 worked examples in the coach prompt** (S effort, L impact): The Best Practices tab says "examples beat rules" but the default `DEFAULT_SYSTEM_COACH` has zero examples. Adding 3 specific, grounded examples (active / dormant / new customer) would immediately improve draft quality across all customers.

5. **#13 — Intent template quick-picks** (S effort, L impact): The intent field is the highest-leverage input the operator controls, but it's blank-slate every time. Pre-baked chips ("Check in", "Missed yesterday", "Post-workout") collapse the intent-to-draft latency from 20s to 3s.

6. **P8 — Onboarding answers in prompt** (S effort, L impact): After a customer answers 6 questions, their goal, age bracket, and style preference are stored but invisible to the AI. Injecting even `goal` and `coachLanguage` would make every post-onboarding draft feel more tailored.

7. **P51 — Graduated urgency scale by days-since-session** (XS effort, L impact): A rules block in the system prompt that tells the model to escalate tone with dormancy. Currently the model decides this on its own and often gets the calibration wrong (either too aggressive with a 2-day absence or too casual with a 14-day absence).

8. **#41 — Structured rating reason categories** (S effort, L impact): Replacing the free-text `prompt()` for 👎 ratings with structured checkboxes makes the rating data actionable. Currently the reasons are in the Firebase log but too varied to aggregate. Structured categories enable the "auto-flag" idea (#44) later.

9. **P88 — Anthropic prompt caching** (M effort, L impact): The system prompt is the same for all customers of the same personality. Adding `cache_control: {type: "ephemeral"}` in the worker saves 90% of input token cost on cached calls. At 100+ API calls/day this compounds quickly.

10. **#31 — "Needs attention" smart sort** (M effort, L impact): A sort mode that scores customers by urgency (dormant + no outbound today = critical; pending draft waiting = high; recently replied = low) would make the morning review workflow dramatically faster. The operator would instantly see the 3 customers who need attention, instead of scanning the full list.
